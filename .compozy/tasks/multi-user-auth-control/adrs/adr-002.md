# ADR-002: Auth Middleware as a Separate Plugin (following the headroom.proxy_extension pattern)

## Status

Accepted

## Date

2026-06-16

## Context

PRD 2 (Authenticated Proxy Gateway) needs to decide where the authentication middleware lives in the architecture. Headroom already has an extension system (`headroom.proxy_extension` entry-point group) and a working example: the `headroom-oauth2` plugin in `plugins/headroom-oauth2/`.

Three approaches were considered: separate plugin, built-in middleware in core, and external service (sidecar).

## Decision

**Plugin middleware as a separate extension** (`plugins/headroom-auth/`), following the `headroom-oauth2` pattern. The plugin registers an entry-point in the `headroom.proxy_extension` group and is activated via `--proxy-extension headroom-auth` or `HEADROOM_PROXY_EXTENSIONS=headroom-auth`. When not activated, the proxy operates exactly as today (zero change to core).

## Alternatives Considered

### Alternative 1: Built-in middleware (directly in `server.py`)

- **Description**: Auth middleware registered directly in `create_app()`, triggered by `--auth` flag
- **Pros**: Less boilerplate (no pyproject.toml, no entry-point, no separate package)
- **Cons**: Couples auth to the proxy core; auth code mixed with proxy code; harder to test in isolation; goes against the established extension pattern
- **Why rejected**: The extension pattern exists precisely to avoid coupling. The `headroom-oauth2` plugin has already proven it works. Putting auth in core would be a step backwards in the architecture.

### Alternative 2: External auth service (sidecar)

- **Description**: Separate HTTP service that validates keys (`POST /auth/verify`). Proxy calls via HTTP before each request.
- **Pros**: Maximum decoupling — auth can be written in another language, scaled independently
- **Cons**: Extra latency (HTTP round-trip per request); one more service to deploy, monitor, and maintain; overengineering for a small-team MVP
- **Why rejected**: Disproportionate operational cost. The extra latency negates part of headroom's benefit (request optimization). Neo4j is already an external service — adding another is unnecessary complexity.

## Consequences

### Positive

- Total isolation — auth disabled = proxy identical to original (zero regression risk)
- Testability — middleware can be tested with a mock Neo4j, without starting the full proxy
- Consistent pattern — any developer who understood `headroom-oauth2` understands `headroom-auth`
- Independent packaging — the plugin can have its own release cycle and dependencies

### Negative

- Plugin boilerplate — pyproject.toml, entry-point, directory structure (about 4 extra files)
- Separate installation — `pip install headroom-ai[auth]` or similar (optional dependency)

### Risks

- **Entry-point discovery**: If the plugin is not installed, the entry-point is not found. Mitigation: clear error at proxy startup (`--proxy-extension headroom-auth` → "plugin 'headroom-auth' not found. Install with: pip install headroom-ai[auth]").
- **Middleware ordering**: Auth must run before rate limiter and memory handler. Mitigation: document the order and test.

## Implementation Notes

- Follows the `install(app: FastAPI, config: ProxyConfig) -> None` signature
- The middleware uses `contextvars` to store `user_id` (pattern: `headroom/proxy/project_context.py`)
- Rate limiter is called within the auth middleware, not as a separate middleware (simplifies ordering)
- Validation cache in memory (TTL 10s), no Redis dependency in MVP
