# ADR-001: Pure Admin CLI (no REST API or Web UI in MVP)

## Status

Accepted

## Date

2026-06-16

## Context

PRD 1 (Admin CLI & User Management) needs to decide how administrators and team leads interact with the system for managing users, roles, teams, proxy API keys, and provider keys. Headroom is a 100% CLI tool today (`headroom proxy`, `headroom wrap`, etc.), operated by developers in a terminal.

Three approaches were considered: pure CLI, CLI + REST API, and Web UI.

## Decision

**Pure CLI** as the sole administration interface in the MVP. The `headroom auth` command exposes Click subcommands that interact directly with Neo4j (without an intermediary proxy). No REST API or web interface will be created for administration.

## Alternatives Considered

### Alternative 1: CLI + REST API

- **Description**: CLI commands hit REST endpoints exposed by the proxy (`POST /admin/users`, etc.)
- **Pros**: Enables future web UI, CLI works remotely without direct Neo4j access
- **Cons**: Doubles effort (API + CLI client), API needs its own authentication (admin key), adds attack surface
- **Why rejected**: Disproportionate effort for the MVP. A REST API can be added later as a natural evolution when there is demand for a web UI or external integrations.

### Alternative 2: Web UI (admin dashboard)

- **Description**: Full web interface for managing users, roles, and keys
- **Pros**: More friendly for non-technical users
- **Cons**: 3x more effort (frontend + API + auth), outside the CLI ecosystem, headroom currently has no web UI at all
- **Why rejected**: Out of scope for the MVP. Can be re-evaluated when the product has adoption in larger teams with non-technical admins.

## Consequences

### Positive

- Faster MVP — reuses Click infrastructure that headroom already dominates
- Zero new dependencies — CLI talks directly to Neo4j (driver already exists in the project)
- Works via SSH — admins don't need an extra exposed port
- Simple security — shell access = command access. No additional HTTP attack surface

### Negative

- Admin/team lead needs shell access to the machine where Neo4j is running (not remote)
- No integration with external tools (CI/CD, webhooks, Slack bots)
- If a Web UI is ever needed, a REST API will need to be created (rework)

### Risks

- **Shell access**: Requires admins to have SSH configured on the server. Mitigation: document and recommend SSH keys, not passwords.
- **Passwords in history**: `set-provider-key` receives the provider key as input — could leak to shell history. Mitigation: read from stdin with prompt or file.

## Implementation Notes

- The CLI uses the same Neo4j driver as `DirectMem0Adapter` (established pattern)
- Commands that receive secrets (`set-provider-key`, `create-key`) must use interactive prompt or stdin, never positional arguments
- The Neo4j database must be accessible from the host where the CLI runs (localhost or internal network)
