# ADR-003: Dual Audit — Neo4j (Structured) + Qdrant (Semantic) via CLI

## Status

Accepted

## Date

2026-06-16

## Context

PRD 3 (Usage Audit & Analytics) needs to decide how to store and query request logs. Two types of queries are needed:

1. **Structured queries**: "how many tokens did João spend this week?", "which model does the backend team use most?", "top 5 users by consumption"
2. **Semantic queries**: "what did the team ask about authentication?", "find requests similar to this error"

Neo4j is ideal for structured queries (graphs, aggregations, relationships). Qdrant is ideal for semantic search (embeddings, similarity). Both are already in docker-compose.yml.

Three approaches were considered: Neo4j + Qdrant (complete), Neo4j only, and Neo4j + Qdrant + Web dashboard.

## Decision

**Dual audit — Neo4j for structured queries + Qdrant for semantic search, accessed via CLI (`headroom audit`).** Each authenticated request generates a `(:RequestLog)` node in Neo4j and an embedding in Qdrant (collection `headroom_request_logs`). Access scope is by role: developer sees their own requests, team lead sees their team's requests, admin sees everything.

## Alternatives Considered

### Alternative 1: Neo4j only (no Qdrant)

- **Description**: Only structured queries in Neo4j, no semantic search
- **Pros**: Simpler — one less collection to manage, fewer embeddings to compute
- **Cons**: Loses semantic search capability ("what did the team ask about?"). If Qdrant is added later, all old requests need to be re-embedded (expensive)
- **Why rejected**: Qdrant is already provisioned and running (for memory). Adding a new collection has low marginal cost. Semantic search is the differentiator that justifies using Qdrant beyond Neo4j.

### Alternative 2: Neo4j + Qdrant + Web dashboard

- **Description**: Same as the decision, but with a web interface for graphs and dashboards
- **Pros**: More visual, better for presenting data to non-technical management
- **Cons**: 3x more effort (frontend); outside headroom's CLI ecosystem; would need a REST API to feed the dashboard
- **Why rejected**: Out of scope for the MVP. Can be added as a future PRD when there is management demand.

## Consequences

### Positive

- Covers both query types: quantitative (Neo4j) and qualitative (Qdrant)
- CLI consistent with PRD 1 (`headroom auth`) and PRD 2 (flags)
- Qdrant is already running — marginal cost of adding a collection is low
- Access scope aligned with roles (admin, team_lead, developer)
- Embeddings are generated asynchronously (do not block the request)

### Negative

- Two data sources to maintain (Neo4j + Qdrant)
- Complexity of buffer/flush: asynchronous writes to two systems

### Risks

- **Embedding cost**: Zero — fastembed runs locally via ONNX. CPU mitigation: embedding only from summary (200 words), can be disabled (`--no-audit-semantic`).
- **Neo4j vs Qdrant divergence**: One write may fail while the other succeeds. Mitigation: Qdrant is best-effort; Neo4j is the source of truth. If Qdrant fails, structured queries still work.

## Implementation Notes

- Embeddings use fastembed BAAI/bge-small-en-v1.5 (384 dims, ONNX local) — same model as relevance scoring, zero added cost
- Async buffer: requests are accumulated in a deque and flushed every 50 entries or 5 seconds
- `(:RequestLog)` has a denormalized `username` field to avoid JOIN in common queries
- Qdrant collection `headroom_request_logs` is created at startup if it doesn't exist
