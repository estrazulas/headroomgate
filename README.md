Headroom compresses everything your AI agent reads вҖ” tool outputs, logs, RAG chunks, files, and conversation history вҖ” before it reaches the LLM. Same answers, fraction of the tokens.

## рҹҡҖ Why this fork

Same upstream compression engine, **plus a complete team gateway** built in:

| You want toвҖҰ | You get |
|-------------|---------|
| **Manage who uses the proxy** | Create users, teams, and API keys вҖ” revoke access anytime |
| **Stop sharing provider keys** | Devs authenticate with their own key; provider keys stay encrypted on the server |
| **Know who spent what** | Structured audit trail: every request logged with user, model, tokens, latency |
| **Find what people asked** | Semantic search вҖ” "show me debugging sessions about memory leaks" |
| **Investigate outliers** | Per-user request history with `--history` вҖ” see exactly what they worked on |
| **Onboard in 2 minutes** | Interactive setup wizard + one-command wrapper for Claude Code |

> Built on [chopratejas/headroom](https://github.com/chopratejas/headroom). All upstream compression features preserved.

## рҹ”җ What's different in this fork

This fork adds **multi-user auth and access control** on top of the compression proxy, turning Headroom into a managed LLM gateway for teams.

### Admin CLI & user management вҶ’ [see how](openspec/changes/archive/2026-06-16-admin-cli-user-management/DUMMY.MD)

- **User registration** вҖ” `headroom auth create-user` creates users with encrypted API keys (Fernet)
- **Team organization** вҖ” users belong to teams; the admin sets roles and rate limits
- **Key rotation** вҖ” `headroom auth create-key <user>` generates a new API key, revoking the old one

### Auth proxy gateway вҶ’ [see how](openspec/changes/archive/2026-06-16-auth-proxy-gateway/DUMMY.MD)

- **Middleware plugin** вҖ” every proxy request is authenticated before reaching the LLM
- **API key validation** вҖ” reads the `Authorization: Bearer <key>` header, looks up the user in Neo4j
- **Rate limiting** вҖ” per-user and per-team token budgets enforced by a token-bucket algorithm
- **Identity propagation** вҖ” the authenticated user identity travels through `contextvars`, available to every downstream component without passing it explicitly

### Usage audit & analytics вҶ’ [see how](openspec/changes/archive/2026-06-16-audit-analytics/DUMMY.MD)

- **Structured logging (Neo4j)** вҖ” every request is recorded: who made it, which model, how many tokens, latency, cache hit/miss
- **Semantic search (Qdrant)** вҖ” request prompts are embedded as vectors so you can search by meaning: *"show me everything about database migrations"*
- **Async buffer** вҖ” writes are batched in memory (50 entries or 5 seconds, whichever comes first), so audit logging adds **zero latency** to the client response
- **CLI analytics** вҖ” `headroom usage summary`, `headroom usage top --by-tokens`, `headroom usage search "architecture patterns"`

### Request history вҶ’ [see how](openspec/changes/archive/2026-06-17-usage-history/DUMMY.MD)

- **Chronological listing** вҖ” `headroom usage user alice --history --last 7d` shows individual requests with timestamp, model, tokens, and summary
- **No guessing required** вҖ” see exactly what a user worked on without crafting semantic search queries

## How it works (layman's version)

Think of Headroom as an **office building** where each LLM request is a visit:

| Component | Technology | Real-world analogy |
|-----------|-----------|-------------------|
| **Auth middleware** | Neo4j + Fernet | The **badge reader** at the turnstile вҖ” only people with a valid badge (API key) get in |
| **Rate limiter** | Token bucket | The **elevator keycard** вҖ” each team has a monthly quota; when it's spent, the elevator stops |
| **Audit store** | Neo4j | The **sign-in book** at reception вҖ” records who came in, when, which room they used, how long they stayed |
| **Semantic log** | Qdrant + embeddings | The **librarian's index cards** вҖ” you can search "who asked about Python threading?" and find matching visits even if the exact words differ |

```bash
pipx install --force \
```

Available extras:

| Extra | Provides |
|-------|----------|
| `proxy` | FastAPI proxy server, MCP tools, ONNX compression |
| `code` | AST-based code compression (tree-sitter) |
| `mcp` | MCP server (`headroom_compress`, `headroom_retrieve`, `headroom_stats`) |
| `memory` | Local vector memory (hnswlib, sqlite-vec) |
| `auth` | User management CLI, API key crypto, Neo4j auth store deps |
| `all` | Everything above |

The **headroom-auth plugin** (separate wheel) adds the auth middleware itself вҖ” entry-point, rate limiter, provider injection, identity propagation.

> **Note:** Auth requires a running Neo4j instance. Set `NEO4J_URI`, `NEO4J_USER`, and `NEO4J_PASSWORD` in your environment.

## When to use В· When to skip

**Use this fork if youвҖҰ**
- run an LLM proxy for a **team** and need to know who is using what
- want **API key authentication** without setting up a separate auth service
- need an **audit trail** вҖ” who asked what, when, and how much it cost
- want **semantic search** over request history ("find all conversations about Kubernetes")
- need **per-user rate limits** so one heavy user doesn't exhaust the team budget

**Skip this fork if youвҖҰ**
- only use Headroom for **personal** projects вҖ” the upstream proxy is simpler
- don't want to run **Neo4j** or **Qdrant** alongside the proxy
- only use a single provider's native compaction and don't need team features
- work in a sandboxed environment where local processes can't run

```bash

git fetch upstream --tags
git merge upstream/main

gh release create "v${VER}" dist/headroom_ai-*.whl dist/headroom_auth-*.whl \
```
